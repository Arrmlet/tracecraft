How many agents can race one bucket for a claim?

A contention benchmark for tracecraft's atomic task claim — and an honest look at why claim latency tracks the number of simultaneously-working agents.

The claim, bounded.

What the data earns: exactly-one-winner held on S3 trials with duplicate wins — verified by re-reading the stored object with a separate client. And claim latency rises with the number of agents only when they fire simultaneously: a staggered control with the same agent counts stays flat at the base round-trip. So the driver is simultaneity, not request count.

What it does not earn: absolute cloud-S3 latency (this is localhost MinIO over loopback — a lower bound); a clean split of server-side arbitration from client-side thread skew at N≥16 (the two co-move, r≥0.8 — shown below, against ourselves); and it is the worst case — agents claiming different steps don't contend at all.

1 · The elbow: claim latency vs simultaneous agents

N agents race the same fresh key at the same instant. Exactly one wins (conditional PUT); the rest get a clean rejection. Median latency stays flat through ~8 agents, then climbs.

2 · The falsification test: simultaneous vs staggered

Same agent counts, same backend, same key — but now agents fire apart instead of together, so no two requests are ever in flight at once. If the curve were about request count, it would still rise. It doesn't.

3 · The mechanism: watch the queue form

One representative trial per N. Each agent is a row: a hollow tick when it arrived (fired its PUT) and a filled dot when the store decided. At N=2 the dots cluster; as N grows the decisions fan out into a staircase while arrivals stay near-vertical — that growing gap is the contended object being resolved one PUT at a time.

4 · Stated against ourselves: the confound

The honest weak point. Our own simultaneity metric — how tightly the barrier released the threads — itself grows with N. At N≥16 it correlates with latency strongly (Pearson r on the right), so at the top of the curve we cannot cleanly separate "the store queued" from "our threads didn't fire perfectly together." That's exactly why §2 and §3 exist.

5 · Losers cost more than winners

A rejected claim (PreconditionFailed — the designed loss path, not an error) returns after the winning PUT is durable, so it sits above the winner at every N. Honest note: there's a fixed ~20–30ms floor on the reject path even at N=2 where there's barely any contention, so most of the loser cost is error-path handling, not queueing.

6 · Correctness control: S3 holds, HuggingFace breaks

Not a latency comparison — a proof the harness can detect a broken coordinator. HuggingFace buckets have no conditional write, so the claim is check-then-write: under contention every agent "wins" and the Hub keeps the last. Same invariant check, opposite result.

7 · The other primitive: do messages survive a burst?

Claiming is a race for one key; messaging is the opposite — every message is its own key, so there's no contention. The risk there isn't speed, it's delivery: if two messages map to the same key, one silently vanishes. We found exactly that, fixed it, and measured both. Below: fired at once.
